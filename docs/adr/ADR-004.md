# ADR-004: Two-tier test strategy for database infrastructure

**Status:** Accepted

## Context

The current acceptance suite runs entirely against in-memory repositories. It is
fast, requires no infrastructure, and fully covers all application behaviour.
Introducing real database infrastructure raises a question: where and how do we
verify that the real repository implementations behave correctly?

Two naive approaches have meaningful drawbacks:

- **Replace in-memory with a real DB everywhere** — tests become slow, require
  infrastructure to run locally, and CI becomes fragile.
- **Duplicate the full acceptance suite against a real DB** — expensive to run,
  maintains the same scenarios twice, and tests application behaviour that is
  already covered.

## Decision

We use two tiers:

**Tier 1 — In-memory acceptance suite (existing)**
HTTP-level tests using `InMemory*` repositories. Covers all application behaviour.
Runs on every commit (pre-commit and CI). No infrastructure required.

**Tier 2 — Repository integration tests (new)**
Tests the real database adapters directly at the repository interface level — not
via HTTP. Each test exercises one repository implementation (e.g.
`TypeOrmProductRepository`) against a real PostgreSQL instance spun up by
Testcontainers. They verify the adapter fulfils the `IProductRepository` /
`IOrderRepository` contract: save, retrieve, handle missing records.

Tier 2 does not repeat application or business logic scenarios — those are owned
by Tier 1. The question Tier 2 answers is: does the real adapter implement the
port correctly?

Tier 2 runs on pre-push and in CI. Developers accept the cost of container
startup (~2-3s) as the price of verifying real infrastructure before integration.

**Open decision — suite isolation mechanism**
How Tier 2 is isolated from the standard suite (separate Jest project, a
`--testPathPatterns` selector, or another approach) is not yet decided. This will
be explored with an experiment before implementation.

## Consequences

- The fast local feedback loop is preserved for the commit cycle — Tier 1 runs
  unchanged on pre-commit.
- Real PostgreSQL behaviour (type coercion, constraints, query semantics) is
  verified on pre-push and CI without duplicating application-level scenarios.
- Testcontainers handles container lifecycle — no persistent local infrastructure
  or pre-configured CI services required.
- The repository interface (`IProductRepository`, `IOrderRepository`) becomes the
  explicit contract under test in Tier 2, reinforcing the port boundary.
- Tier 2 tests are fewer and cheaper to maintain than a full HTTP-level suite
  against a real DB.
- Pre-push hook becomes slower. Accepted trade-off.
