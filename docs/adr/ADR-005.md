# ADR-005: Use DATABASE_URL presence to select persistence implementation, with in-memory restricted to non-production environments

## Status

Accepted

## Context

The application has two persistence implementations: in-memory repositories (dev and test only) and TypeORM repositories (targeting real Postgres). Both implement the same domain port interfaces. The app needs a mechanism to select which implementation to wire at startup without requiring code changes between environments.

In a production-ready system, accidentally running with in-memory persistence must not be possible — data loss would be silent and undetectable.

## Decision

At module initialisation, apply the following rules:

- `NODE_ENV=production` and `DATABASE_URL` absent → **throw at startup**. Do not allow the app to run in production without a database.
- `DATABASE_URL` present (any environment) → wire TypeORM repositories.
- `DATABASE_URL` absent and `NODE_ENV` ≠ `production` → wire in-memory repositories (dev and test default).

## Consequences

- **Production safety**: a misconfigured production environment fails loudly at startup rather than silently serving ephemeral data.
- The in-memory path remains the default for dev and all automated tests — no test setup changes required.
- `NODE_ENV` must be set correctly in production deployments — if omitted, the app will not fail safe.
- Every new aggregate must follow the same conditional wiring pattern in its module — a convention that must be communicated to contributors.
- The two modes have different durability guarantees; acceptance tests run against in-memory and do not verify TypeORM behaviour (covered by integration tests per ADR-004).
