# ADR-003: Use London school during TDD, then refactor use case tests to Classic school

**Status:** Accepted

## Context

London school (outside-in) TDD drives object design effectively: mocking
collaborators forces explicit interfaces and reveals coupling early. However,
once the design has stabilised, interaction-based tests become a liability —
they break on refactoring even when observable behaviour is unchanged, because
they assert on *how* objects collaborate rather than *what* the system does.

Classic school (Detroit/Chicago) tests assert on state and outcomes. They are
more durable under refactoring but provide weaker design pressure during the
initial TDD cycle.

## Decision

We use both in sequence, per layer:

**During RED → GREEN:** use London school. Mock all collaborators. Let the
interactions drive the design and surface coupling.

**On GREEN, as the final refactor step:** replace mocks in use case tests with
real in-memory repository implementations. Assert on state — what was persisted,
what was returned — rather than on which methods were called.

This applies to **use case tests only**. Controller tests remain London school:
controllers are thin adapters whose only job is delegation, so verifying the
delegation *is* the test.

The in-memory repositories used here are the same ones used in acceptance tests
— no new infrastructure is introduced.

## Consequences

- Use case tests survive internal refactoring (renaming methods, restructuring
  commands) as long as observable behaviour is preserved.
- The design pressure of London school is retained where it matters most: during
  the initial cycle, before the design has settled.
- Use case tests acquire a light integration flavour (real repository, real domain
  objects). This is a conscious trade-off, not a mistake.
- Controller tests remain interaction-based — verifying delegation is their purpose.
- The acceptance tests remain the primary regression safety net throughout.
