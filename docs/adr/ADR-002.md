# ADR-002: Omit formal interfaces for driving-side ports

**Status:** Accepted

## Context

In pure Hexagonal Architecture, both sides of the boundary have ports. The driven
side (right) has repository interfaces that the application calls through. Symmetrically,
the driving side (left) should have use case interfaces that controllers call through,
with the concrete use case class as the adapter. This would mean defining e.g.
`IPlaceOrderUseCase` alongside `PlaceOrderUseCase`, with controllers depending on
the interface.

## Decision

We do not define formal interfaces for use cases. Controllers depend directly on
the concrete use case classes.

## Consequences

- Less ceremony: one fewer file per use case.
- No loss of testability: Jest can mock concrete classes directly; an interface is
  not required for unit-testing controllers.
- Acceptable trade-off: use cases are unlikely to have multiple implementations
  or need to be swapped independently of each other.
- If a use case ever needs an alternate implementation, an interface can be
  introduced at that point (YAGNI).
