# ADR-001: Use Clean Architecture layering inside a Hexagonal boundary

**Status:** Accepted

## Context

Two well-established patterns apply to this codebase: Hexagonal Architecture
(Ports and Adapters, Alistair Cockburn) and Clean Architecture (Robert C. Martin).
They share the same core principle — the dependency rule: outer layers depend on
inner layers, never the reverse — but differ in scope. Hexagonal defines the
boundary pattern between the application and the outside world. Clean Architecture
defines the internal layering within that boundary.

## Decision

We use both together.

Clean Architecture provides the internal layers:

- `domain` — pure business logic, entities, invariants. No framework dependencies.
- `application` — use cases. Orchestrate domain via repository interfaces.
- `infrastructure` — all external concerns: HTTP, persistence, NestJS wiring.

Hexagonal Architecture provides the boundary mechanism:

- **Driven ports** — repository interfaces defined in `domain` (e.g. `IProductRepository`).
  The application calls out through these; infrastructure provides the implementations.
- **Driving adapters** — HTTP controllers in `infrastructure/http`. They translate
  external requests into use case calls.

The dependency rule is enforced statically by dependency-cruiser.

## Consequences

- The domain layer is pure TypeScript with zero framework dependencies — highly
  testable and portable.
- Swapping infrastructure concerns (e.g. replacing in-memory persistence with a
  database) requires only changing the module binding.
- Contributors familiar with either pattern will find the structure recognisable.
